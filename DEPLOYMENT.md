# EasyCart — Deployment Guide

One repo, two layers: the Django application (repo root) and the zero-trust
AWS infrastructure (`terraform/`). One pipeline deploys both.

```
repo/
├── easycart_project/  store/          <- Django app
├── infra/
│   ├── lambda/<fn>/lambda_function.py <- REAL Lambda source (terraform deploys these)
│   └── config.json                    <- generated by scripts/generate_config.py
├── terraform/                         <- all IaC (modules + usecase stacks)
├── scripts/
│   ├── build_lambdas.sh               <- infra/lambda/<fn>/ -> <fn>.zip
│   └── generate_config.py             <- terraform outputs -> infra/config.json
└── .github/workflows/                 <- easycart-deploy.yml, terraform-destroy.yml
```

## One-time setup

1. **Terraform state bucket** (shared by laptop + CI):
   ```bash
   aws s3 mb s3://easycart-tfstate-<unique> --region us-east-1
   aws s3api put-bucket-versioning --bucket easycart-tfstate-<unique> \
     --versioning-configuration Status=Enabled
   grep -rl "easycart-tfstate-mel4821" . | xargs sed -i 's/easycart-tfstate-mel4821/easycart-tfstate-<unique>/'
   ```

2. **Migrate your laptop's local state to S3** — your current infra was
   applied with local state; CI must see it or it will try to re-create
   everything and collide. In EACH stack directory you have applied
   (networking, storage, database, app, frontend):
   ```bash
   cd terraform/usecase/networking   # repeat per applied stack
   terraform init -migrate-state -backend-config=../../backend.hcl   # answer "yes"
   ```
   (Copy your existing terraform.tfstate files into the matching
   terraform/usecase/<stack>/ folders first if you are starting from a
   fresh clone of this repo.)

3. **GitHub secrets** (Settings → Secrets → Actions):
   `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_SESSION_TOKEN`.
   Learner Lab credentials expire per session — re-paste all three each
   session or the pipeline fails with `ExpiredToken`.

## What a push to main does

1. Builds the 5 Lambda zips from `infra/lambda/<fn>/` (your real handlers)
2. Applies terraform: networking → storage → database → app → frontend
3. Regenerates `infra/config.json` from live terraform outputs
   (new Cognito pool/client/secret, bucket, SNS ARN, EB names)
4. Zips the Django app (excluding env/, terraform/, library local/, .git)
5. Uploads the bundle to the private S3 bucket and deploys it to the
   Elastic Beanstalk environment, waiting until Ready

PRs stop at validate + networking plan; nothing is applied.

## After first deploy — seed the data (once)

The DynamoDB tables and bucket start empty. From a machine with AWS creds,
after `generate_config.py` has produced a fresh config.json:
```bash
python3 infra/setup_products.py        # products + images into DynamoDB/S3
python3 infra/EasyCartAdminUsers.py    # admin user(s)
```

## Verifying (there is no public URL — by design)

The EB environment is internal-only. Reach it through the hub via SSM:
```bash
# find the EB instance id (tag: environment name), then:
aws ssm start-session --target <instance-id> \
  --document-name AWS-StartPortForwardingSession \
  --parameters '{"portNumber":["80"],"localPortNumber":["8080"]}'
# then browse http://localhost:8080
```
